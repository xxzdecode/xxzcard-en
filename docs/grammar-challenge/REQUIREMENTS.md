## 1. 模块定位

- 语法挑战是学生端独立模块，不影响现有单词系统。
- 保留统一流程：`首页入口 → 练习页面 → 结果反馈`。
- 视觉采用“轻互动数字测验”，不直接套用课堂即时练习的火车、魔法等玩法模板。
- 内容来源于老师已经教授并发布在老师端“随堂练习”中的语法内容，但挑战题难度明显提高。

## 3. 后续生成逻辑

- 不自动读取“最新随堂练习”并生成挑战。
- 每次由老师明确告诉系统本次教授了什么内容、需要生成哪一天的语法挑战。
- 生成并确认后，再将该份练习加入学生端“语法挑战”模块。
- 每份挑战必须保留独立日期、标题、知识点、题目内容和结果反馈配置，不能覆盖历史练习。

## 4. 待实现需求：练习存档与手动切换

### 4.1 需求描述

老师需要能够保存每一份历史语法挑战，并在之后手动决定学生端当前使用哪一份练习。

示例：

- 已存档 `2026-07-15｜句子骨架`；
- 已存档 `2026-07-16｜代词 + be 动词`；
- 老师可以在管理入口中把任意一份设置为“当前启用”；
- 学生端首页的“语法挑战”只进入老师当前启用的练习；
- 切换练习不删除或覆盖其他练习。

### 4.3 初步技术方向

优先采用“练习内容文件化存档 + 数据库只保存目录和当前启用状态”的混合方案：

- 每份题目和页面配置作为独立 JSON / JS 数据文件或独立练习目录进入 Git，便于版本管理、回退和长期存档；
- Supabase 保存轻量目录信息，例如练习 ID、日期、标题、知识点、状态、当前启用项；
- 学生答题记录按练习 ID 保存，避免切换练习后历史成绩错位；
- 不建议把完整练习页面和全部题目直接塞入现有 `main` 数据对象，避免单词系统数据结构继续膨胀并相互影响。

该技术方向只是预案，正式实施前需结合本仓库当前 Supabase 数据结构和页面代码再确认。
