# 英语单词卡生成与导入规则书

## 1. 使用目的

本规则用于生成英语单词卡的标准导入数据。

适用场景：

- ChatGPT 根据单词列表批量生成单词卡
- Codex / 前端项目读取固定 JSON 字段
- 后续 Supabase 或本地 JSON 批量导入
- 给小学生 / 初中基础薄弱学生学习使用

核心原则：

1. 信息足够有用，但不要太复杂。
2. 字段固定，不随便新增变量。
3. 内容面向小学生和基础薄弱学生，解释要短、清楚、可理解。
4. 只写真正有教学价值的信息，不为了填满字段而硬编。
5. 生成结果必须可以直接作为导入数据使用。

---

## 2. 标准 JSON 字段

每张单词卡固定使用以下字段：

```json
{
  "word": "",
  "meaning": "",
  "pos": "",
  "phonetic": "",
  "emoji": "",
  "morphology": [],
  "collocations": [],
  "irregularForms": [],
  "synonyms": [],
  "wordFamily": [],
  "tip": ""
}
```

字段顺序建议固定为：

1. `word`
2. `meaning`
3. `pos`
4. `phonetic`
5. `emoji`
6. `morphology`
7. `collocations`
8. `irregularForms`
9. `synonyms`
10. `wordFamily`
11. `tip`

---

## 3. 字段说明

### 3.1 word

英文单词本体。

要求：

- 动词默认用原形，如 `go`、`eat`、`play`
- 名词默认用单数，如 `child`、`box`、`apple`
- 形容词默认用原级，如 `happy`、`big`
- 不写复数、三单、过去式作为主词条，除非这个词本身就是独立高频词

示例：

```json
"word": "wind"
```

---

### 3.2 meaning

中文核心意思。

要求：

- 尽量短
- 先写最常用意思
- 不写太多解释
- 多义词只写当前教学阶段最需要的意思

示例：

```json
"meaning": "风"
```

如果确实需要多个意思，可以用中文分号隔开：

```json
"meaning": "跑；经营"
```

但小学生阶段尽量避免一次塞太多意思。

---

### 3.3 pos

词性。

固定使用以下缩写：

| 英文词性 | 推荐写法 |
|---|---|
| 名词 | `n.` |
| 动词 | `v.` |
| 形容词 | `adj.` |
| 副词 | `adv.` |
| 介词 | `prep.` |
| 连词 | `conj.` |
| 代词 | `pron.` |
| 限定词 / 冠词 | `det.` |
| 感叹词 | `interj.` |

统一使用缩写形式，不要混用 `noun`、`verb`、`adjective` 等完整写法，避免前端筛选不统一。

示例：

```json
"pos": "n."
```

---

### 3.4 phonetic

音标。

要求：

- 使用 IPA 音标
- 用 `/ /` 包起来
- 优先选择常见教材可接受读音
- 不需要同时写英式和美式，除非之后项目专门扩展
- 多音词如果当前词义只对应一个读音，只写当前词义读音

示例：

```json
"phonetic": "/wɪnd/"
```

注意：

- `wind` 作“风”时读 `/wɪnd/`
- `wind` 作“缠绕、上发条”时可读 `/waɪnd/`
- 小学生卡片里只写当前意思对应读音，避免混乱

---

### 3.5 emoji

单词对应的小图标。

要求：

- 一个 emoji 即可
- 尽量直观
- 不要使用太抽象或容易误解的 emoji
- 如果没有特别合适的，可以选接近场景的 emoji

示例：

```json
"emoji": "🌬️"
```

---

## 4. morphology 构词 / 前后缀知识

### 4.1 字段结构

固定使用：

```json
"morphology": [
  {
    "part": "",
    "meaning": "",
    "knowledgeKey": ""
  }
]
```

不要新增字段，如 `base`、`word`、`type`，除非前端以后确定支持。

每一条 `morphology` 只表示一个具体的构词知识点。

如果一个单词同时包含多个构词知识点，就拆成多条分别写。

例如：

```json
"morphology": [
  {
    "part": "un-",
    "meaning": "表示不、相反：fortunate → unfortunate",
    "knowledgeKey": "prefix-negative"
  },
  {
    "part": "-ly",
    "meaning": "加在形容词后，常变成副词：unfortunate → unfortunately",
    "knowledgeKey": "suffix-adverb"
  }
]
```

不要把多个前后缀混在同一条里写：

```json
"morphology": [
  {
    "part": "un- / -ly",
    "meaning": "表示否定和副词",
    "knowledgeKey": "prefix-negative, suffix-adverb"
  }
]
```

这种写法不推荐，因为后续前端无法准确按分类整理。

---

### 4.2 使用范围

`morphology` 只写当前单词本体里真实存在、且有教学价值的构词信息，例如：

- 常见前缀：`un-`、`re-`、`in-`、`dis-`
- 常见后缀：`-y`、`-er`、`-ful`、`-less`、`-ly`
- 常见词根：只在非常简单、学生能理解时使用
- 词形变化中能帮助记忆的部分

`morphology` 的目的不是完整词源分析，而是帮助学生记单词、理解单词变化，并为后续“构词专项区”提供分类索引。

---

### 4.3 不要硬写

如果一个词没有明显适合小学生理解、且当前单词本体真实包含的前缀、后缀或词根，就写空数组：

```json
"morphology": []
```

不要为了填字段而拆错词，也不要把派生词拓展硬写进当前词的 `morphology`。

例如：

- `wind` 本体没有适合拆解的前缀或后缀，所以写：

```json
"morphology": []
```

不要写：

```json
"morphology": [
  {
    "part": "-y",
    "meaning": "加在名词后，常变成形容词",
    "knowledgeKey": "suffix-adjective"
  }
]
```

因为 `wind` 本身并不包含 `-y`。

`morphology` 只负责记录当前单词本身的构词知识，方便后续前端根据 `knowledgeKey` 做专项索引。

---

### 4.4 knowledgeKey 用途

`knowledgeKey` 用来给后续“构词专项区”做分类索引。

它不是展示给学生看的主要内容，而是前端用来归类的标签。

以后前端可以根据 `knowledgeKey` 自动整理出专项页面。

例如：

```text
表示否定
├─ un-
│  ├─ unhappy
│  ├─ unlucky
│  └─ unfortunate
├─ in-
│  ├─ incorrect
│  └─ inactive
├─ dis-
│  ├─ disagree
│  └─ dislike
```

也就是说，前端不应该只靠搜索单词拼写来判断前后缀，而应该优先读取 `morphology` 里的结构化数据。

例如：

```js
word.morphology.some(item => item.knowledgeKey === "prefix-negative")
```

这样可以避免误判。

例如：

- `sun` 里有 `un`，但不是 `un-` 否定前缀
- `under` 以 `un` 开头，但不是 `un-` 否定前缀
- `dish` 以 `dis` 开头，但不是 `dis-` 否定前缀
- `water` 以 `er` 结尾，但不是表示“人 / 职业”的 `-er`

所以 `knowledgeKey` 必须和当前这一条 `morphology` 对应，不要写成整张单词卡的总分类。

---

### 4.5 knowledgeKey 建议

常用 `knowledgeKey` 可以先固定为：

| 类型 | knowledgeKey |
|---|---|
| 表否定 / 相反的前缀 | `prefix-negative` |
| 表重复 / 再次的前缀 | `prefix-again` |
| 形容词后缀 | `suffix-adjective` |
| 名词后缀 | `suffix-noun` |
| 人 / 职业后缀 | `suffix-person` |
| 副词后缀 | `suffix-adverb` |
| 比较级 / 最高级 | `suffix-comparison` |
| 拼写变化 | `spelling-change` |

后续如需更多分类，再统一扩展，不要临时乱写。

---

### 4.6 多个构词点的写法

如果一个单词里有多个有教学价值的构词点，就分别写成多条。

例如 `unfortunately`：

```json
"morphology": [
  {
    "part": "un-",
    "meaning": "表示不、相反：fortunate → unfortunate",
    "knowledgeKey": "prefix-negative"
  },
  {
    "part": "-ly",
    "meaning": "加在形容词后，常变成副词：unfortunate → unfortunately",
    "knowledgeKey": "suffix-adverb"
  }
]
```

这样以后前端可以把它同时归入：

- 表否定 / 相反：`un-`
- 副词后缀：`-ly`

一个单词可以出现在多个构词专项里，这是允许的。

## 5. collocations 固定搭配 + 例句

### 5.1 字段结构

固定使用：

```json
"collocations": [
  {
    "phrase": "",
    "example": ""
  }
]
```

### 5.2 内容要求

每个搭配包含：

- `phrase`：英文固定搭配或高频自然搭配
- `example`：英文例句 + 中文翻译

格式固定为：

```json
"example": "The strong wind blew my hat away. / 大风把我的帽子吹走了。"
```

### 5.3 搭配数量

建议：

- 普通单词：尽量 1 个搭配
- 高频动词 / 多用法词：2–3 个搭配
- 不要超过 3 个，避免卡片太重
- 如果没有自然、适合小学生的搭配，可以写空数组

没有合适搭配时：

```json
"collocations": []
```

### 5.4 搭配选择原则

优先选择：

1. 小学生能理解的
2. 高频、自然的
3. 能体现这个词真实用法的
4. 句子短，方便跟读
5. 可以复用到口语或写作里

不要选择：

- 太成人化的句子
- 太抽象的搭配
- 生词太多的例句
- 为了高级而高级的表达

### 5.5 例句难度

例句建议控制在：

- 5–10 个英文单词左右
- 句型尽量简单
- 不使用复杂从句
- 中文翻译自然，不要生硬直译

推荐：

```json
{
  "phrase": "strong wind",
  "example": "The strong wind blew my hat away. / 大风把我的帽子吹走了。"
}
```

不推荐：

```json
{
  "phrase": "strong wind",
  "example": "The strong wind broke the window. / 大风打破了窗户。"
}
```

原因：中文“打破窗户”不自然，更自然是“把窗户吹破了”。

## 6. irregularForms 特殊形式

### 6.1 字段结构

固定使用数组：

```json
"irregularForms": []
```

如果有内容，推荐写成对象：

```json
"irregularForms": [
  {
    "label": "过去式",
    "form": "go → went"
  }
]
```

前端展示时可以直接显示为：

```text
过去式：go → went
```

不要写得太复杂，不需要额外解释句。

---

### 6.2 推荐 label 写法

常用 `label` 可以先固定为：

| 类型 | label |
|---|---|
| 现在式 | `现在式` |
| 过去式 | `过去式` |
| 过去分词 | `过去分词` |
| 第三人称单数 | `三单` |
| 复数 | `复数` |
| 比较级 | `比较级` |
| 最高级 | `最高级` |

例如：

```json
"irregularForms": [
  {
    "label": "现在式",
    "form": "be → am / is / are"
  },
  {
    "label": "过去式",
    "form": "be → was / were"
  },
  {
    "label": "复数",
    "form": "mouse → mice"
  },
  {
    "label": "比较级",
    "form": "good → better"
  },
  {
    "label": "最高级",
    "form": "good → best"
  }
]
```

前端展示为：

```text
现在式：be → am / is / are
过去式：be → was / were
复数：mouse → mice
比较级：good → better
最高级：good → best
```

---

### 6.3 只写真正特殊的形式

只写不规则变化，例如：

```json
"irregularForms": [
  {
    "label": "过去式",
    "form": "go → went"
  }
]
```

```json
"irregularForms": [
  {
    "label": "三单",
    "form": "have → has"
  }
]
```

```json
"irregularForms": [
  {
    "label": "复数",
    "form": "child → children"
  }
]
```

```json
"irregularForms": [
  {
    "label": "复数",
    "form": "mouse → mice"
  }
]
```

```json
"irregularForms": [
  {
    "label": "比较级",
    "form": "good → better"
  },
  {
    "label": "最高级",
    "form": "good → best"
  }
]
```

不要写规则变化，例如：

- `wind → winds`
- `play → played`
- `book → books`
- `small → smaller / smallest`

除非当前课程专门讲某个规则变化，例如比较级规则，否则规则变化不放进 `irregularForms`。

---

### 6.4 多个特殊形式的写法

如果一个词有多个特殊形式，就分成多条写。

例如 `be`：

```json
"irregularForms": [
  {
    "label": "现在式",
    "form": "be → am / is / are"
  },
  {
    "label": "过去式",
    "form": "be → was / were"
  }
]
```

例如 `go`：

```json
"irregularForms": [
  {
    "label": "过去式",
    "form": "go → went"
  },
  {
    "label": "过去分词",
    "form": "go → gone"
  }
]
```

一个单词可以有多条特殊形式，前端展示时每条单独换行。

展示效果：

```text
过去式：go → went
过去分词：go → gone
```

---

### 6.5 没有特殊形式时

统一写：

```json
"irregularForms": []
```

不要写奇怪空格、换行符或注释。

---

## 7. synonyms 近义词 / 相关词

### 7.1 字段结构

固定使用：

```json
"synonyms": [
  {
    "word": "",
    "meaning": ""
  }
]
```

字段名保持 `synonyms` 不变，避免影响现有前端和导入逻辑。

但前端展示时，不一定写“近义词”，可以显示为：

```text
相关词
```

或：

```text
近义 / 相关词
```

因为这个字段里可以放真正近义词，也可以放适合学生理解的相关词。

---

### 7.2 内容原则

这里可以放：

- 真正近义词
- 对小学生有帮助的相关词
- 同一语义范围内的词

但不要强行放难词。

例如 `wind`：

```json
"synonyms": [
  {
    "word": "breeze",
    "meaning": "微风；轻轻的风"
  }
]
```

注意：`breeze` 不是完全等于 `wind`，但作为相关词可以保留。

---

### 7.3 数量

建议 0–2 个。

没有合适的就写：

```json
"synonyms": []
```

---

## 8. wordFamily 词族

### 8.1 字段结构

固定使用：

```json
"wordFamily": [
  {
    "word": "",
    "pos": "",
    "meaning": ""
  }
]
```

### 8.2 pos 写法

`wordFamily` 里的 `pos` 也使用统一缩写：

- `n.`
- `v.`
- `adj.`
- `adv.`
- `prep.`
- `conj.`
- `pron.`
- `det.`
- `interj.`

示例：

```json
"wordFamily": [
  {
    "word": "teacher",
    "pos": "n.",
    "meaning": "老师"
  }
]
```

---

### 8.3 内容原则

只放和主词明显相关、学生能理解的词。不要放太简单的词，比如unhappy->happy。

优先：

- `happy → happiness`
- `teach → teacher`
- `play → player`
- `help → helpful`

不要放太远、太难、太学术的派生词。

---

### 8.4 数量

建议 0–3 个。

没有合适的写：

```json
"wordFamily": []
```

---

## 9. tip 小知识 / 易错点

### 9.1 字段类型

固定为字符串：

```json
"tip": ""
```

---

### 9.2 内容要求

`tip` 只写一句短提示。

可以写：

- 易错读音
- 易混词
- 常见用法
- 简单文化小知识
- 记忆提醒

不要写：

- 长篇解释
- 太高级的语法
- 和当前词无关的内容
- 模糊的“这是一个常用词”废话
- 已经拆过的前后缀

推荐：

```json
"tip": "wind 是名词时读 /wɪnd/，表示“风”。"
```

如果没有值得写的内容，可以写空字符串：

```json
"tip": ""
```

---

## 10. 输出格式要求

### 10.1 批量生成时

如果用户给多个单词，默认输出 JSON 数组：

```json
[
  {
    "word": "",
    "meaning": "",
    "pos": "",
    "phonetic": "",
    "emoji": "",
    "morphology": [],
    "collocations": [],
    "irregularForms": [],
    "synonyms": [],
    "wordFamily": [],
    "tip": ""
  }
]
```

---

### 10.2 单个单词时

如果用户只给一个单词，也推荐输出 JSON 数组，方便批量导入。

单个单词也使用同样结构：

```json
[
  {
    "word": "",
    "meaning": "",
    "pos": "",
    "phonetic": "",
    "emoji": "",
    "morphology": [],
    "collocations": [],
    "irregularForms": [],
    "synonyms": [],
    "wordFamily": [],
    "tip": ""
  }
]
```

不要在示例里写具体单词，避免示例内容影响后续生成。

---

### 10.3 不要输出解释

当用户说“生成单词卡 / 按导入格式生成”时，只输出合法 JSON。

不要加解释，不要加 Markdown 标题，不要加代码块标记，除非用户明确要求“检查 / 说明 / 修改建议”。

---

## 11. 内容生成总原则

### 11.1 面向对象

目标学生：

- 小学生
- 初中基础薄弱学生
- 拼读、词汇、句子能力都需要重建的学生

所以内容要：

- 简单
- 高频
- 可读
- 可跟读
- 可造句
- 不堆知识点

---

### 11.2 不要乱编

如果不确定某个词的：

- 音标
- 不规则变化
- 词族
- 近义词
- 前后缀

宁可留空或写简单信息，不要编复杂内容。

---

### 11.3 不要过度模块化

一张卡片里最重要的是：

1. 认识单词
2. 会读
3. 知道中文意思
4. 会一个常见搭配
5. 能看懂一个短句

其他信息都是辅助。

---

### 11.4 字段为空时的写法

数组字段统一写空数组：

```json
"morphology": []
```

```json
"collocations": []
```

```json
"irregularForms": []
```

```json
"synonyms": []
```

```json
"wordFamily": []
```

字符串字段没有内容时写空字符串：

```json
"tip": ""
```
