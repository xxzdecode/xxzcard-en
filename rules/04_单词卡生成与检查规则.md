# 04_单词卡生成与检查规则

更新时间：2026-07-11  
状态：正式流程规则  
适用范围：英语单词卡生成、旧数据增强、批量处理、格式检查与文件交付

> 本文件规定单词卡从“原始单词”到“可直接导入文件”的完整工作流程。  
> 字段内容标准以《02_单词卡字段与内容规则》为准。  
> 最终文本格式以《03_单词卡真实导入格式》为准。

---

## 1. 核心原则

1. Supabase 是正式单词库的唯一准确数据源。
2. 不再把 `list.txt` 作为日常去重依据。
3. 去重在生成前通过独立去重工具完成。
4. 独立去重工具只读取 Supabase，不写入、不修改、不删除数据。
5. ChatGPT / Work 只处理已经确认需要生成的新词。
6. 现有网站导入器只负责解析和导入，不承担日常去重任务。
7. 不要求为了本流程修改学生网站或现有导入器。
8. 生成完成后必须分别进行内容检查和格式检查。
9. 最终直接交付可导入文件，不在聊天中粘贴大量数据。
10. 任务过大时自动拆批，全部完成后再合并和复查。

---

## 2. 标准工作流

```text
原始输入
→ 清理空格、标点、粘连和疑似拼写错误
→ 识别单个单词、多词表达和普通搭配
→ 抽取核心词
→ 拆分需要独立学习的普通搭配
→ 保留“原始表达 → 核心词”的对应关系
→ 对标准化后的核心词进行批次内部去重
→ 独立去重工具查询 Supabase
→ 区分已有核心词、新核心词和异常项
→ 生成增强版内容
→ 将原始表达写入对应核心词的 collocations
→ 检查字段内容
→ 转换为真实导入格式
→ 检查解析格式
→ 输出 TXT 文件
→ 用户手动导入现有网站

```

各环节职责必须分开，不能混成一个工具。

---

## 3. 第一步：原始单词整理

原始输入可能来自：

- 课堂记录
- 图片或截图
- Word / PDF 笔记
- 用户手动输入
- 旧单词表
- 旧 JSON
- 已有单词卡文件

处理前需要完成基础整理：

- 去除前后空格
- 统一用于比较的小写形式
- 清理多余逗号、斜杠和重复标点
- 识别明显粘连
- 识别疑似拼写错误
- 保留最终展示时的标准拼写

以下内容不能直接自动确认：

- `gless`
- `r d`
- `key/cold`
- 含数字
- 多个单词粘连
- 词组和单词混在一起
- 无法区分音标、中文和英文的内容

疑似错误必须标记为“需要人工确认”，不能擅自修正后直接生成。
### 3.1 核心词标准化

完成基础清理后，必须在查询 Supabase 和生成单词卡之前进行核心词标准化。

处理结果应区分：

1. 单个核心词
2. 短语动词及其核心动词
3. 固定表达及其核心实词
4. 需要拆分的普通搭配
5. 可以保留为完整主词条的特殊多词词项
6. 无法确定核心词、需要人工确认的表达

标准化示例：

```text
figure out → figure
as well → well
wind direction → wind + direction
average temperature → average + temperature
solve problem → solve + problem
```

标准化时必须同时保留原始表达，供后续写入对应核心词的 `collocations`。

Supabase 查询必须使用标准化后的核心词，不能只查询未经处理的原始短语。

例如输入：

```text
figure out
```

应查询：

```text
figure
```

不能只查询：

```text
figure out
```

输入：

```text
wind direction
```

应分别查询：

```text
wind
direction
```

如果 Supabase 已经存在核心词，不得因为输入的是相关短语而重新创建一张多词卡片。

例如 Supabase 已经存在 `figure` 时：

* 不创建 `word: figure out`
* 将其标记为已有核心词
* 如需增加 `figure out`，应进入已有单词卡更新流程
* 不在新词生成批次中重复生成 `figure`

## 4. 第二步：独立去重工具

去重必须通过独立工具完成，不绑定学生网站。

独立工具负责：

1. 检查当前输入内部重复。
2. 读取 Supabase 中已有的 `word` 字段。
3. 使用标准化形式比较。
4. 区分：
   - 需要生成的新词
   - Supabase 已有词
   - 当前输入内部重复词
   - 需要人工确认的异常项
5. 支持复制新词列表。
6. 支持下载新词 TXT。

比较时默认执行：

```text
trim()
toLowerCase()
```

以下形式视为同一个词：

```text
Apple
apple
 apple
apple 
```

独立工具禁止：

- 写入 Supabase
- 修改已有单词
- 删除单词
- 自动覆盖内容
- 生成音标或释义
- 生成单词卡
- 修改学生网站
- 修改现有导入器

---

## 5. Supabase 的职责

Supabase 负责保存正式单词卡数据。

它是以下信息的唯一准确来源：

- 单词是否已经存在
- 当前正式单词卡内容
- 当前单词总数量
- 已导入批次的最终结果
- 后续旧词修改后的最新版本

项目记忆可以记录：

- 最近处理了哪一批
- 某批大约新增多少词
- 哪些词等待修正

但项目记忆不能替代 Supabase 保存完整词库。

不再要求人工长期维护多个版本的：

```text
list.txt
list(1).txt
new-list.txt
final-list.txt
updated-list.txt
```

如需要词表备份，可以从 Supabase 临时导出，但导出文件不是主数据源。

## 6. 第三步：生成任务输入

交给 ChatGPT / Work 的输入必须是：

> 已经通过独立去重工具确认，需要生成的新词。

推荐输入方式：

```text
增强版：
apple
book
chair
```

或：

```text
生成增强版单词卡：
apple、book、chair
```

如输入中包含：

- 英式 IPA
- 中文义
- 当前课程词义
- 词性
- 特殊教学要求

应优先保留用户明确提供的信息，但仍需检查是否准确。

生成阶段仍应进行一次本批次内部重复检查，防止复制或处理中再次产生重复。

---

## 7. 第四步：增强版内容生成

生成时必须遵守《02_单词卡字段与内容规则》。

默认要求：

- 使用固定 11 个字段。
- 字段顺序固定。
- 全部使用英式 IPA。
- 使用适合学生的核心中文义。
- 每个词提供 1–5 个有教学价值的搭配。
- 搭配数量由实际教学价值决定。
- 每个搭配必须配对应例句。
- 有真实构词知识时填写 `morphology`。
- 有真正不规则形式时填写 `irregularForms`。
- 有真正近义词时填写 `synonyms`。
- 有真实、常用词族时填写 `wordFamily`。
- 有易混点时填写 `tip`。
- 没有合适内容时保留空数组或空字符串。
- 不为了填满字段而编造内容。

生成不是机械补字段。

需要重点判断：

- 搭配是否自然
- 例句是否真正使用搭配
- 中文是否自然
- 词族关系是否真实
- 近义词是否真正接近
- tip 是否重复 morphology
- irregularForms 是否误写规则变化

---

## 8. 旧数据迁移

旧数据迁移不能只做字段名替换。

### 8.1 先检查原数据

检查：

- 文件能否解析
- 是否存在旧字段
- 是否缺字段
- 是否存在空值类型错误
- 音标是否混入美式形式
- 内容是否密度过低
- 搭配是否自然
- 例句是否对应
- 词族是否错误
- tip 是否重复
- irregularForms 是否包含规则变化

### 8.2 字段转换

旧字段统一转换到：

```text
word
meaning
pos
phonetic
emoji
morphology
collocations
irregularForms
synonyms
wordFamily
tip
```

禁止在新文件中继续输出：

```text
en
zh
ex
note
```

### 8.3 内容重新审核

旧数据迁移默认视为一次完整内容审核。

不能只保证“能导入”，还必须检查：

- 内容是否达到当前增强版标准
- 搭配是否有教学价值
- 音标是否为英式
- 各字段是否真实
- 字段之间是否重复

---

## 9. 大批量任务拆分

当单词数量较多时，不应一次性生成全部内容。

推荐顺序：

```text
按日期拆分
→ 日期内按小批拆分
→ 每批生成
→ 每批内容检查
→ 每批格式检查
→ 合并
→ 合并后再次检查
```

拆分原则：

- 优先按原始日期或来源拆分。
- 每批数量以能够完整审核为准。
- 不能因为数量大而降低内容密度。
- 不能先生成全部再跳过逐词检查。
- 中间文件和最终文件必须区分。

最终合并前检查：

- 批次之间是否重复
- 字段顺序是否一致
- 空行是否正确
- 是否混入旧格式
- 是否有批次缺失
- 是否有临时说明混入

---

## 10. 内容检查

生成完成后，先进行内容检查。

### 10.1 基础字段

- [ ] `word` 拼写正确。
- [ ] `meaning` 与当前教学义一致。
- [ ] `pos` 使用统一缩写。
- [ ] `phonetic` 为英式 IPA。
- [ ] `emoji` 合理或为空字符串。

### 10.2 morphology

- [ ] 当前词真实包含该构词结构。
- [ ] 没有牵强拆词。
- [ ] `knowledgeKey` 合理。
- [ ] 没有与 `tip` 重复。

### 10.3 collocations

- [ ] 数量为 1–5 个。
- [ ] 每个搭配自然、常见、有教学价值。
- [ ] 多个搭配不重复同一用法。
- [ ] 每个例句真正使用对应搭配。
- [ ] 英文和中文语义一致。
- [ ] 难度适合学生。

### 10.4 irregularForms

- [ ] 使用 `label` 和 `form`。
- [ ] 只写不规则变化。
- [ ] 没有普通复数、规则过去式、规则三单或规则比较级。

### 10.5 synonyms

- [ ] 只放真正近义或高度接近的词。
- [ ] 没有普通主题相关词。
- [ ] 数量合理。

### 10.6 wordFamily

- [ ] 词族关系真实。
- [ ] 内容常用且适合学生。
- [ ] 没有拼写相似但意义无关的词。

### 10.7 tip

- [ ] 有真实易混点时已经填写。
- [ ] 没有重复其他字段。
- [ ] 内容简短、具体。
- [ ] 没有无意义说明。

### 10.8 核心词与多词表达

- [ ] `word` 是否为正确的核心词。
- [ ] 短语动词是否选择了主要动词。
- [ ] 含功能词的固定表达是否选择了承担主要词义的实词。
- [ ] 普通搭配是否按需要拆成独立单词。
- [ ] 原始表达是否写入对应核心词的 `collocations`。
- [ ] `meaning`、`pos` 和 `phonetic` 是否对应核心词本身。
- [ ] 同一核心词是否在当前批次中被重复生成。
- [ ] Supabase 去重是否使用标准化后的核心词。
- [ ] 已存在核心词是否被错误地重新创建为多词卡片。
- [ ] 含空格的 `word` 是否属于已经确认的合理例外。
---

## 11. 格式转换

内容检查通过后，按照《03_单词卡真实导入格式》转换。

要求：

- 使用逐字段 `key: value`
- 每个字段单独占一行
- 11 个字段顺序固定
- 数组字段保持单行 JSON
- 空数组写为 `[]`
- 空 tip 写为空字符串
- 卡片之间空一行
- 不加外层 JSON 数组
- 不加每张卡外层对象
- 不加 Markdown 代码框
- 不加标题或说明

转换阶段不能修改已经确认的字段内容。

---

## 12. 格式检查

格式转换后必须检查：

### 字段

- [ ] 每张卡都有 11 个字段。
- [ ] 字段名完全正确。
- [ ] 字段顺序正确。
- [ ] 没有旧字段。
- [ ] 空数组字段没有被省略。

### 数组

- [ ] 数组全部在同一行。
- [ ] 使用标准英文双引号。
- [ ] JSON 对象结构合法。
- [ ] 没有多余逗号。
- [ ] 没有括号缺失。

### 卡片分隔

- [ ] 卡片内部没有空行。
- [ ] 卡片之间有一个完整空行。
- [ ] 文件开头没有说明。
- [ ] 文件结尾没有聊天文字。

### 外层格式

- [ ] 没有外层数组。
- [ ] 没有每张卡外层 `{}`。
- [ ] 没有 Markdown 代码框。
- [ ] 没有 JSONL。

### 文件

- [ ] 文件编码为 UTF-8。
- [ ] 文件能正常打开。
- [ ] 文件内容不为空。
- [ ] 文件名符合当前任务要求。

## 13. 最终交付

用户要求生成文件时，必须实际生成成品。

默认交付：

- 可直接导入的 `.txt` 文件

根据任务需要，可同时交付：

- 重复词说明
- 已跳过词列表
- 批次统计
- 合并版文件
- ZIP 压缩包
- 内容检查报告
- 格式检查报告

最终交付方式按照核心词标准化、拆分、合并和去重后的卡片数量判断：

- 最终卡片数量达到 15 张或以上时，默认直接生成并交付可导入的 `.txt` 文件，不在聊天中粘贴完整导入文本。
- 最终卡片数量少于 15 张时，可以直接在聊天中提供导入文本。
- 用户明确要求文件时，无论数量多少，都应直接生成文件。
- 数量以最终生成的核心词卡片数为准，不以原始输入项数量或空格分词数量为准。

回复应简要说明：

- 处理了多少词
- 是否拆批
- 是否发现异常
- 是否完成内容检查
- 是否完成格式检查
- 提供文件下载链接

---

## 14. 网站导入阶段

现有网站导入器保持原有职责：

- 读取导入文本
- 解析字段
- 显示或写入单词卡
- 返回导入成功或失败

本流程不要求：

- 修改网站导入器
- 在导入器中增加去重
- 修改学生网站页面
- 修改 Supabase 表结构
- 增加数据库唯一约束
- 增加新的后台模块

用户在生成前已经通过独立工具完成去重。

---

## 15. 失败处理

遇到以下情况时，不能直接交付为“完成”：

- 单词拼写无法确认
- 音标无法确认
- 输入包含疑似粘连
- 真实导入格式不确定
- 数组无法解析
- 字段结构与规则冲突
- 某批内容明显密度不足
- 合并后出现重复或缺失
- 文件无法打开

处理方式：

1. 标记具体问题。
2. 保留已完成且确认正确的部分。
3. 不猜测不确定内容。
4. 修正后重新检查。
5. 只有通过检查的文件才作为最终交付。

## 16. Chat、Work 与 Codex 分工

### Chat

适合：

- 讨论规则
- 处理少量单词
- 审核某些字段
- 判断搭配、词族和易混点
- 讨论异常词

### Work

适合：

- 批量生成单词卡
- 处理多个文件
- 拆批和合并
- 完成内容检查
- 完成格式检查
- 输出最终文件

### Codex

适合：

- 制作独立 Supabase 去重工具
- 修复解析器
- 编写本地校验脚本
- 修改代码项目
- 运行自动化检查

普通单词卡内容生成不需要交给 Codex。

---

## 17. 与其他规则文件的关系

### 《02_单词卡字段与内容规则》

负责：

- 字段含义
- 字段结构
- 内容质量
- 英式 IPA
- 搭配、词族、近义词和 tip

### 《03_单词卡真实导入格式》

负责：

- 最终逐字段格式
- 数组序列化
- 空行
- 外层包装限制
- 文件编码

### 本文件

负责：

- 从原始单词到最终文件的工作流程
- Supabase 去重职责
- 批量处理
- 内容检查
- 格式检查
- 文件交付

发生冲突时：

1. 内容问题以《02》为准。
2. 排版和解析问题以《03》为准。
3. 执行顺序和检查流程以本文件为准。
