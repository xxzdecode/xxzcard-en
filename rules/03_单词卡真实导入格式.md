# 03_单词卡真实导入格式

更新时间：2026-07-11  
状态：正式格式规则  
适用范围：英语单词卡导入文本生成、旧数据转换、批量导入前检查

> 本文件只规定最终导入文本的排版、字段顺序和序列化方式。  
> 字段内容质量、搭配数量、英式 IPA、词族、近义词和易混提示等规则，以《02_单词卡字段与内容规则》为准。

---

## 1. 导入格式的核心原则

英语单词卡的真实导入文本不是普通 JSON 数组，也不是 JSONL。

每张卡使用逐字段格式：

```text
word: ...
meaning: ...
pos: ...
phonetic: ...
emoji: ...
morphology: ...
collocations: ...
irregularForms: ...
synonyms: ...
wordFamily: ...
tip: ...
```

每张卡之间必须保留一个完整空行。

导入文本中不得加入：

- Markdown 代码框
- 外层 JSON 数组
- 每张卡外层对象
- 标题
- 说明文字
- 编号
- 项目符号
- 聊天回复内容
- 与导入数据无关的注释

---

## 2. 固定字段顺序

每张单词卡固定按照以下顺序输出：

1. `word`
2. `meaning`
3. `pos`
4. `phonetic`
5. `emoji`
6. `morphology`
7. `collocations`
8. `irregularForms`
9. `synonyms`
10. `wordFamily`
11. `tip`

完整模板：

```text
word: 
meaning: 
pos: 
phonetic: 
emoji: 
morphology: []
collocations: []
irregularForms: []
synonyms: []
wordFamily: []
tip: 
```

字段顺序不得随意调整。

## 3. 字段名要求

必须使用：

```text
word
meaning
pos
phonetic
emoji
morphology
collocations
irregularForms
synonyms
wordFamily
tip
```

禁止使用旧字段：

```text
en
zh
ex
note
```

禁止自行更改大小写，例如：

```text
Word
Meaning
wordfamily
irregularforms
```

字段名必须与当前项目标准完全一致。

---

## 4. 基础字段写法

基础字符串字段采用：

```text
字段名: 内容
```

例如：

```text
word: book
meaning: 书
pos: n.
phonetic: /bʊk/
emoji: 📖
tip: book 作名词是“书”；不要把 look 和 book 看混。
```

规则：

- 冒号使用英文半角冒号 `:`
- 冒号后保留一个空格
- 每个字段单独占一行
- 不在字段值外加多余引号
- 不在行末加逗号
- 不在行末加分号
- 不使用制表符对齐
- 不把两个字段写在同一行

正确：

```text
word: book
meaning: 书
```

错误：

```text
word:"book",
meaning:"书",
```

错误：

```text
word: book meaning: 书
```

---

## 5. 数组字段写法

以下字段必须使用 JSON 数组形式：

- `morphology`
- `collocations`
- `irregularForms`
- `synonyms`
- `wordFamily`

每个数组必须完整写在同一行。

### 5.1 空数组

没有内容时统一写：

```text
morphology: []
collocations: []
irregularForms: []
synonyms: []
wordFamily: []
```

不要省略字段。

不要写：

```text
morphology:
```

不要写：

```text
morphology: null
```

不要写：

```text
morphology: ""
```

---

### 5.2 非空数组

数组内部使用标准 JSON 对象格式。

例如：

```text
morphology: [{"part":"un-","meaning":"表示不、相反：happy → unhappy","knowledgeKey":"prefix-negative"}]
```

```text
collocations: [{"phrase":"feel unhappy","example":"I feel unhappy today. / 我今天不开心。"},{"phrase":"look unhappy","example":"She looks unhappy. / 她看起来不开心。"}]
```

```text
irregularForms: [{"label":"过去式","form":"go → went"},{"label":"过去分词","form":"go → gone"}]
```

```text
synonyms: [{"word":"sad","meaning":"难过的"}]
```

```text
wordFamily: [{"word":"happiness","pos":"n.","meaning":"幸福；快乐"}]
```

规则：

- 数组必须写在一行。
- 对象字段名使用英文双引号。
- 字符串值使用英文双引号。
- 多个对象之间使用英文逗号分隔。
- 数组外使用 `[` 和 `]`。
- 对象外使用 `{` 和 `}`。
- 不在数组中加入换行。
- 不在数组后加逗号。
- 不使用中文引号。
- 不使用单引号代替双引号。

---

## 6. 单词卡之间的空行

每张单词卡之间必须保留一个完整空行。

正确：

```text
word: first
meaning: 第一个
pos: adj.
phonetic: /fɜːst/
emoji: 1️⃣
morphology: []
collocations: [{"phrase":"the first time","example":"This is my first time here. / 这是我第一次来这里。"}]
irregularForms: []
synonyms: []
wordFamily: []
tip: 

word: second
meaning: 第二个
pos: adj.
phonetic: /ˈsekənd/
emoji: 2️⃣
morphology: []
collocations: [{"phrase":"the second floor","example":"My classroom is on the second floor. / 我的教室在二楼。"}]
irregularForms: []
synonyms: []
wordFamily: []
tip: 
```

错误：

```text
tip: 
word: second
```

如果单词之间没有空行，解析器可能无法正确识别卡片边界。

规则：

- 卡片内部不插入空行。
- 卡片之间保留一行空白。
- 文件开头不需要空行。
- 文件结尾可以有或没有一个换行符，但不要附加其他文本。

---

## 7. 禁止外层包装

真实导入文本不能写成普通 JSON 数组。

错误：

```json
[
  {
    "word": "book",
    "meaning": "书"
  }
]
```

也不能给每张卡增加外层对象：

```text
{
word: book
meaning: 书
}
```

也不能使用 JSONL：

```text
{"word":"book","meaning":"书"}
{"word":"apple","meaning":"苹果"}
```

也不能放入 Markdown 代码框：

````text
```text
word: book
meaning: 书
```
````

最终交付的导入文件中，只保留实际可导入文本。

## 8. 特殊字符与文本要求

### 8.1 中文和英文标点

基础字段中的中文解释可以使用中文标点。

数组内部必须保持合法 JSON：

- 使用英文双引号
- 使用英文逗号
- 使用英文冒号
- 使用英文方括号和花括号

### 8.2 例句分隔

搭配例句固定使用：

```text
英文例句 / 中文翻译
```

英文与中文之间使用：

```text
 / 
```

即斜杠两侧各保留一个空格。

### 8.3 双引号

数组字段的字符串值中如需出现引号，应避免直接使用未转义的英文双引号。

优先改写句子，避免产生解析风险。

例如不要直接写：

```text
example":"He said "hello". / 他说“你好”。"
```

可以改为：

```text
example":"He said hello to me. / 他向我问好。"
```

### 8.4 换行

任何单个字段值都不得自行换行。

尤其禁止：

- 一个 `collocations` 数组拆成多行
- 一个 `morphology` 对象拆成多行
- 一个例句中间换行
- 一个 `tip` 拆成多行

---

## 9. 文件编码与文件类型

建议使用：

- 文件类型：`.txt`
- 文件编码：UTF-8
- 换行格式：常规 Windows 或 Unix 换行均可，但同一文件应保持一致

文件中不得混入：

- BOM 引发的异常字符
- 富文本格式
- Word 隐藏样式
- 网页复制产生的不可见字符
- 制表符
- 自动编号

如从 Word、网页或聊天复制，应在导入前检查是否产生全角符号或不可见字符。

## 10. 已验证的格式示例

以下示例用于说明真实排版结构：

```text
word: book
meaning: 书
pos: n.
phonetic: /bʊk/
emoji: 📖
morphology: []
collocations: [{"phrase":"read a book","example":"I read a book every day. / 我每天读一本书。"},{"phrase":"a story book","example":"This is a story book for children. / 这是一本儿童故事书。"}]
irregularForms: []
synonyms: []
wordFamily: []
tip: book 作名词是“书”；不要把 look 和 book 看混。
```

该示例主要证明：

- 字段顺序
- 逐字段格式
- 数组单行
- 字段值不加外层引号
- 无外层 JSON 数组

正式生成内容时，仍需同时遵守《02_单词卡字段与内容规则》。

---

## 11. 含不规则形式的标准示例

```text
word: go
meaning: 去；走
pos: v.
phonetic: /ɡəʊ/
emoji: 🚶
morphology: []
collocations: [{"phrase":"go home","example":"I go home at five. / 我五点回家。"},{"phrase":"go to school","example":"We go to school every day. / 我们每天去上学。"},{"phrase":"go shopping","example":"Mum and I go shopping on Saturday. / 我和妈妈星期六去购物。"}]
irregularForms: [{"label":"过去式","form":"go → went"},{"label":"过去分词","form":"go → gone"}]
synonyms: []
wordFamily: []
tip: go 后接 home 时不加 to；说 go to school 时需要加 to。
```

说明：

- `irregularForms` 使用 `label` 和 `form`
- 非空数组保持单行
- 多个对象使用英文逗号连接
- `go` 使用英式音标 `/ɡəʊ/`

该示例需要在当前项目解析器中完成一次实际导入测试。测试通过后，可标记为“已验证示例”。

## 12. 导入前格式检查清单

每次生成导入文件后，必须检查：

### 字段

- [ ] 每张卡都有 11 个固定字段。
- [ ] 字段名称完全正确。
- [ ] 字段顺序正确。
- [ ] 没有旧字段 `en / zh / ex / note`。
- [ ] 没有缺失空数组字段。

### 排版

- [ ] 每个字段单独占一行。
- [ ] 冒号后有一个空格。
- [ ] 卡片内部没有空行。
- [ ] 卡片之间有一个完整空行。
- [ ] 文件中没有标题、编号或说明。

### 数组

- [ ] 所有数组都在单行内完成。
- [ ] 数组使用标准 JSON 双引号。
- [ ] 没有中文引号。
- [ ] 没有多余逗号。
- [ ] 没有对象或数组括号缺失。
- [ ] 空内容统一使用 `[]`。

### 外层结构

- [ ] 没有外层 JSON 数组。
- [ ] 没有每张卡的外层 `{}`。
- [ ] 没有 Markdown 代码框。
- [ ] 没有 JSONL 格式。

### 文件

- [ ] 文件编码为 UTF-8。
- [ ] 文件可以正常打开。
- [ ] 文件内容不是空的。
- [ ] 没有不可见字符或富文本格式。
- [ ] 已使用当前解析器进行实际测试。

## 13. 解析器优先级

格式判断优先级：

1. 当前项目 `import.js` 的实际解析逻辑
2. 已经在真实页面中导入成功的示例
3. 本文件中的格式规则
4. 历史聊天中的旧说明

如果本文件与当前解析器行为冲突：

- 先停止批量生成。
- 检查解析器是否已经更新。
- 确认真实可导入格式。
- 再修订本文件。
- 不得在未确认时同时保留两套格式。
