## 1. 项目范围

本项目包括：

- 英语教学规划与每日备课
- 英语单词卡内容生成与审核
- 单词卡真实导入格式批量生成与检查
- 英式 IPA 与音标训练维护
- 独立 HTML 互动练习
- Word 练习与校内笔记整理
- 英语提升计划网站的整体 UI 系统
- 与以上内容相关的方案讨论和 Codex 执行准备

本项目暂不把以下内容作为固定规则：

- 尚未定稿的奖励系统
- 当前临时开发待办
- 当前学生进度
- 当前已完成内容
- 最近批次记录
- 当前数据库统计
- 临时页面方案
- 具体页面尚未确认的布局

这些动态内容通过项目内最新状态记录或具体任务聊天管理。

## 2. 稳定规则与动态状态

项目内容分为两类。

### 2.1 稳定规则

稳定规则放在项目文件中。

例如：

- 单词卡字段
- 内容质量
- 导入格式
- 英式 IPA
- HTML 默认交付形式
- iPad 横屏和手机竖屏适配
- 网站整体 UI 原则
- 分层教学原则

稳定规则不应因为一次临时任务随意修改。

---

### 2.2 动态状态

动态状态不写进稳定规则文件。

例如：

- 音标已经完成系统教学
- 当前正在学习什么语法
- 某个学生最近掌握了什么
- 最近一批新增了哪些单词
- 当前网站正在改哪个页面
- 某个功能是否已经上线
- 临时待办和近期问题

动态状态以项目内最新记录为准。

建议使用固定聊天：

```text
教学与项目状态更新
```

状态更新建议格式：

```text
状态更新｜日期

1. 当前已完成：
2. 当前正在进行：
3. 仍不稳定：
4. 下一阶段：
5. 替代此前哪条旧状态：
```

新状态应明确替代旧状态，避免项目同时引用互相冲突的历史描述。

### 代码修改

方案讨论可以在本项目中完成。

真实代码执行由 Codex 完成。

执行代码任务时，执行卡只写目标项目标识，不写本地绝对路径。项目标识优先使用：

- GitHub 仓库标识，例如 `owner/repository`；
- 已确认且能够唯一识别的项目名称；
- 必要时补充预期的关键文件或模块名称。

Codex 应在当前运行环境中定位实际仓库，并通过以下信息核对项目身份：

- Git remote 的 `owner/repository`；
- 仓库名称；
- 适用的 `AGENTS.md`；
- 执行卡列出的关键文件或模块。

本地目录可能因设备、工作区或运行环境变化而不同。只要项目身份能够唯一确认，本地绝对路径与历史记录不同不得作为阻塞理由。

只有在以下情况才应阻塞：

- 当前环境中找不到目标项目；
- 存在多个候选项目且无法唯一确认；
- Git remote、项目名称或关键文件明显冲突；
- 继续执行可能修改错误仓库。

发生阻塞时，应列出已检查的候选项目和冲突原因，等待用户确认。不得因为历史执行卡中的旧路径不存在，就直接忽略当前环境中身份明确的正确仓库；也不得在身份未确认时擅自选择相似目录。

Codex 的具体工作方式以当前工作区和目标项目中实际适用的 `AGENTS.md` 为准。本项目不另外维护重复的 Codex 实现规则。

## 7. 常用口令

### “生成导入格式”

默认表示：

- 先识别单个单词、多词表达和普通搭配。
- 短语动词选择主要动词作为核心词。
- 含功能词的固定表达选择承担主要词义的实词作为核心词。
- 普通搭配按需要拆成独立单词。
- 原始表达保留在对应核心词的 `collocations` 中。
- 使用标准化后的核心词查询 Supabase。
- 同一核心词只生成一张卡片。
- 按《02_单词卡字段与内容规则》生成完整且有教学价值的内容。
- 按《03》输出真实导入格式。
- 最终卡片达到 15 张或以上时，直接交付可导入 TXT 文件。
- 最终卡片少于 15 张时，可以在聊天中直接提供文本；用户明确要求文件时仍直接生成文件。

---

### “生成 HTML”

默认表示：

- 完整独立 HTML
- CSS 和 JavaScript 内联
- iPad 横屏优先
- 手机竖屏适配
- 儿童友好
- 优先点击、配对、分类和选择
- 不只依赖拖拽
- 直接交付 `.html` 文件

---

### “整理英语校内笔记”

默认表示：

- 生成结构化 Markdown 主文档
- 按需要生成 CSV 或 JSON
- 多单元任务可拆分
- 最后检查和合并
- 需要时生成完整压缩包

---

### “出执行卡”

默认表示：

- 只整理已经确认的方案，不重新展开背景，不擅自新增需求。
- 生成一张完整、可独立执行的 Markdown 执行卡。
- 执行卡直接上传到 GitHub 仓库：

  ```text
  xxzdecode/codex-task-queue
  ```

- 新任务默认上传到：

  ```text
  pending/NNN-简短任务名称.md
  ```

- 不在仓库根目录放置待执行卡。
- 编号使用三位数字，例如 `001`、`002`、`003`。
- 编号是执行卡的主要识别依据，文件名不包含日期。
- 新编号应检查 `pending/`、`running/` 和 `blocked/` 中已使用的编号，并使用当前最大编号加 1；已用编号原则上不重复。
- 任务名称只概括主要目标，尽量控制在 6–12 个字。
- 执行时以编号为准，例如：`执行任务 001`。

执行卡 YAML 头部至少包含：

- `id`
- `title`
- `status`
- `created`
- `target_project`

`target_project` 只写能够识别目标的项目名或仓库标识，例如：

```yaml
target_project: 英语与单词卡系统
```

或：

```yaml
target_project: xxzdecode/xxzcard-en
```

禁止在执行卡中写入本地绝对路径，例如：

```text
D:\xxz-work\xxzcard-en
D:\xxz-work\projects\xxzcard-en
/home/user/projects/xxzcard-en
```

如果 GitHub 仓库标识已知，优先使用稳定的 `owner/repository`。如果没有代码仓库，只写能够唯一识别的项目名称。

Codex 执行时必须在当前环境中定位实际项目，并通过 Git remote、仓库名称、适用的 `AGENTS.md` 和关键文件核对身份：

- 实际本地路径与历史环境不同，但项目身份一致：继续执行，不得阻塞。
- 找不到唯一项目、存在多个冲突候选或仓库身份不一致：转入阻塞并说明需要确认的项目，不要求用户确认某个历史绝对路径。
- 不得仅因执行卡中未提供绝对路径而阻塞。
- 不得在项目身份不明确时擅自选择相似目录。

执行卡必须包含：

1. 任务目标；
2. 修改范围；
3. 每个目标文件的准确仓库相对路径或文件名；
4. 具体章节或插入位置；
5. 操作类型：新增、替换或删除；
6. 需要删除或替换的原内容；
7. 可以直接写入文件的完整新内容；
8. 不需要修改的文件；
9. 禁止事项；
10. 检查要求；
11. 验收标准；
12. 修改后的记忆处理建议；
13. Codex 最终回复格式。

能够确定的修改必须直接写完整，不把整理工作重新交给用户。

执行卡还必须遵守：

- 真实代码结构以目标仓库当前内容为准。
- 不虚构不存在的函数名、类名、仓库相对路径或内部架构。
- 不替 Codex 预写整套实现，但必须把业务规则、数据边界、交互要求和验收标准写清楚。
- 只允许修改执行卡明确列出的文件，不顺手重构或扩大范围。
- 保留未涉及的正确内容。
- 检查过但未修改的文件，不得列为已修改文件。

上传成功后，聊天中只提供：

- 一句简短结论；
- GitHub 执行卡路径；
- 需要用户手动重新提交到 ChatGPT 项目的正式源文件；
- 必要的动态状态迁移或记忆清理说明。

如果 GitHub 上传失败，必须明确说明失败，不得声称已经上传。

### “记忆巡检”

用户发送：

```text
记忆巡检
```

默认直接执行一次完整巡检，不再询问输出格式。

巡检时应对照：

1. 当前能够看到的本项目记忆；
2. 本项目当前正式规则文件；
3. 最新项目状态记录；
4. 当前对话中提供的规划、更新和补充要求；
5. 已被新内容替代的历史状态或旧规则。

只以本轮能够确认的最新文件和最新状态为准。旧对话、旧文件、旧状态和旧结论不得因为仍然可见而自动恢复。

巡检需要识别：

- 应新增到正式文件的稳定内容；
- 应替换或更新正式文件旧内容的内容；
- 已重复、过期、冲突或无需继续保存的内容；
- 应继续作为项目动态状态保存的内容；
- 真正需要作为跨项目长期记忆保留的内容；
- 文件中的名称错误、章节错误、格式错误、职责冲突和重复规则。

动态状态不得写入稳定规则文件。稳定规则已经写入正式文件后，不应继续保留重复的项目记忆副本。

巡检完成后：

1. 生成一张完整 Codex 执行卡；
2. 按执行卡编号与命名规则上传到 `xxzdecode/codex-task-queue/pending/`；
3. 聊天中只提供一句简短结论、GitHub 文件位置、需要重新提交的正式源文件，以及必要的动态状态迁移说明；
4. 不生成额外下载文件，除非用户明确要求；
5. 不直接删除项目记忆、历史聊天或项目文件。

信息不足时：

- 先完成能够确认的部分；
- 在执行卡中明确标记无法确认的项目；
- 不猜测原文；
- 不虚构章节或文件路径；
- 不因为少量不确定内容而拒绝整个巡检。

## 8. Supabase 单词库原则

Supabase 是正式单词库的唯一准确数据源。

日常流程：

```text
原始单词
→ 独立去重工具查询 Supabase
→ 取得新词
→ ChatGPT / Work 生成
→ 用户导入现有网站
```

项目记忆不负责保存完整词库。

不再长期维护多个版本的 `list.txt` 作为主数据源。

现有网站导入器不承担日常去重任务，也不要求为此修改代码。

## 9. 冲突处理优先级

发生冲突时，按以下顺序判断：

1. 用户当前任务中的明确要求
2. 项目内最新动态状态
3. 对应专项规则文件
4. 本总说明
5. 已确认的实际示例
6. 历史聊天
7. 旧项目文件

专项问题以专项文件为准。

例如：

- 内容问题以《02》为准
- 导入排版以《03》为准
- 生成流程以《04》为准
- 音标维护以《05》为准
- HTML 文件生成以《06》为准
- 网站整体视觉以《07》为准

旧规则不得因为仍存在历史聊天而自动恢复。

---

## 10. 项目文件管理原则

项目中只保留最新、正式、仍在使用的规则文件。

不要同时上传：

- 旧完整记忆
- 旧规划书
- 新拆分规则
- 多个相互冲突版本

原“英语”项目和“生成整理”项目可以保留为旧档案。

新项目不需要迁移所有历史聊天。

建议迁移：

- 正式规则文件
- 必要的已验证示例
- 仍有效的参考页面
- 最新动态状态

不建议迁移：

- 失败生成记录
- 已废弃格式
- 重复讨论
- 已解决 bug 排查过程
- 已被替代的 UI 方案

- 是否真的需要单独拆分
- 是否属于动态状态

不为临时需求不断增加规则文件。

## 12. 权威来源、状态迁移与记忆清理

### 12.1 不同信息的权威来源

不同类型的信息分别以下列来源为准：

- 真实代码结构、当前实现和实际行为：以当前目标代码仓库及实际运行结果为准。
- 单词卡、词库是否存在、正式数据和相关统计：以 Supabase 当前数据为准。
- 长期稳定的项目规则：以当前正式规则文件为准。
- 当前教学进度、页面调整状态、近期计划和短期待办：以最新动态状态记录为准。

聊天描述、旧截图、旧导出和历史状态不能覆盖更高优先级的当前来源。

无法读取真实代码或实时数据时，必须明确说明，不能假装已经检查。

### 12.2 状态记录只保存当前有效状态

状态记录负责保存仍会变化的信息，例如：

- 当前教学进度；
- 学生近期掌握情况；
- 当前正在调整的页面；
- 尚未定稿的具体布局；
- 最近批次、短期待办和当前问题；
- 某项功能当前是否完成或上线。

状态记录不重复保存已经进入正式规则文件的稳定规则。

当旧状态对话过长时，可以新建状态对话。新对话只迁移当前仍然有效的完整状态快照，不复制全部历史讨论、失败过程和已被替代方案。

建议命名：

```text
教学与项目状态更新｜阶段 N
```

建议开头格式：

```text
状态更新｜日期

1. 当前已完成：
2. 当前正在进行或学习：
3. 仍不稳定：
4. 下一阶段：
5. 替代此前哪条旧状态：
```

### 12.3 旧状态对话删除条件

删除旧状态对话前，必须确认：

1. 当前仍有效的动态状态已经迁移到新的状态记录；
2. 稳定规则已经写入对应正式规则文件；
3. 修改后的正式源文件已经重新提交到 ChatGPT 项目；
4. 新文件内容和新状态快照已经核对无误；
5. 旧对话中没有仍然唯一存在的重要信息。

满足以上条件后，旧状态对话才可以删除。

删除旧状态对话不会代替正式文件修改，也不能依赖删除操作自动消除规则冲突。

### 12.4 记忆清理边界

记忆巡检可以提出：

- 删除；
- 更新；
- 保留；
- 改为动态状态；
- 已由正式文件接管、等待删除重复记忆。

巡检本身不得直接删除项目记忆。

只有用户确认 Codex 已完成修改、用户已重新提交修改后的正式源文件、并已核对新文件内容后，才能根据用户明确要求处理记忆清理。

Codex 不操作 ChatGPT 项目记忆，不删除历史聊天。
