# 06_HTML与教学材料生成规范

更新时间：2026-07-15  
状态：正式规则  
适用范围：英语互动 HTML 练习、Word 练习、校内笔记整理及配套教学材料生成

> 本文件规定英语教学材料的默认交付形式、视觉风格、交互方式和设备适配要求。  
> HTML 专项练习的主要使用设备为：**iPad 横屏**和**手机横屏**。  
> 手机浏览器仍处于竖屏状态时，默认使用横屏页面旋转兜底，不另做一套竖屏练习逻辑。

---

## 1. 总体原则

1. 所有材料应适合小学生或英语基础薄弱的初中生。
2. 内容难度以当前教学进度和最新状态记录为准。
3. 不擅自加入尚未学习的知识点。
4. 页面和练习优先考虑易读、易点、易理解。
5. 互动练习优先于静态网页。
6. 用户要求文件时，必须实际生成成品文件。
7. 不在聊天中粘贴大量 HTML、Word 内容或长篇数据。
8. HTML、Word 和校内笔记整理分别遵守本文件对应章节。
9. 已确认的视觉和交互规则默认直接使用，不必每次重新讨论。
10. 专项练习的核心操作区必须优先保证在同一屏内完整呈现。
11. iPad 与手机原则上共用同一套横屏布局和同一套题型逻辑。
12. 当前任务中的明确要求高于本文件的默认规则。

# 第一部分：独立 HTML 互动练习

## 2. HTML 默认交付形式

当用户说：

- “生成 HTML”
- “生成 html”
- “给我 HTML”
- “做成网页”
- “做成交互练习”

默认生成一个可以独立运行的完整 HTML 文件。

必须包含：

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>练习标题</title>
  <style>
    /* 内联 CSS */
  </style>
</head>
<body>
  <!-- 页面内容 -->
  <script>
    // 内联 JavaScript
  </script>
</body>
</html>
```

默认要求：

- CSS 全部内联。
- JavaScript 全部内联。
- 不依赖主项目的 `styles.css`。
- 不依赖主项目的其他 JavaScript。
- 双击 HTML 文件即可打开。
- 如非必要，不依赖外部图片、字体、音频或网络资源。
- 直接交付 `.html` 文件。
- 不默认只给源码文本。
- 独立打开和嵌入主站 iframe 时都应保持正常布局。

## 3. 主要设备与横屏策略

HTML 专项练习的适配优先级固定为：

1. **iPad 横屏**
2. **手机横屏**
3. 电脑端正常显示

专项练习原则上只维护一套横屏页面结构，不再为手机竖屏单独制作另一套分类、配对或选择逻辑。

### 3.1 iPad 横屏

iPad 横屏是第一设计和验收基准。

布局要求：

- 充分利用横向空间。
- 优先使用左右分栏、两列卡片、四宫格分类区和双列配对区。
- 核心练习区应在一屏内完整显示。
- 学生完成主要操作时不需要上下滑动页面。
- 标题、知识摘要、进度和按钮应紧凑排列。
- 结果提示不能遮住题目，也不能把练习区向下顶出屏幕。
- 按钮、词卡和目标区应适合手指点击。
- 不使用只适合鼠标悬停的操作。
- 页面不能因为横向空间较大而堆得过密。

建议 CSS 适配范围：

```css
@media (min-width: 768px) and (orientation: landscape) {
  /* iPad 横屏及类似平板横屏布局 */
}
```

推荐布局：

- 左侧素材，右侧操作目标
- 问题与回答左右对应
- 四个分类区使用 `2 × 2`
- 顶部紧凑知识条
- 中间练习台占据剩余空间
- 底部固定高度反馈条

不推荐：

- 单列长页面
- 大段玩法说明
- 每个知识点单独占一行
- 词卡区和分类区分布在页面上下两端
- 需要长距离上下滚动
- 多层大面板互相嵌套

---

### 3.2 手机横屏

手机横屏直接复用 iPad 横屏的页面骨架和交互逻辑。

允许根据屏幕高度进入紧凑模式：

- 标题字号略小。
- 顶部知识条控制在一至两行。
- 按钮减少到主要操作。
- 卡片内边距和区域间距适当缩小。
- 每轮项目数量根据可用高度减少。
- 长句允许换行，但不能把核心操作区挤出屏幕。
- 主要点击区域仍应保持清楚和易点。

适配手机横屏时，应优先根据可用高度调整，而不是只根据屏幕宽度判断。

建议：

```css
@media (orientation: landscape) and (max-height: 520px) {
  /* 手机横屏紧凑模式 */
}
```

不得为了塞入更多内容而：

- 把字号缩得难以阅读。
- 把按钮缩得难以点击。
- 让卡片严重重叠或溢出。
- 恢复成需要上下滑动的长页面。

内容过多时，应减少单轮题量，通过“换一组”“下一轮”等方式覆盖全部内容。

---

### 3.3 手机竖屏状态与旋转兜底

手机物理设备仍应横着拿。若浏览器或系统没有自动切换到横屏，页面默认继续按横屏尺寸绘制，并通过 CSS 旋转整个练习根容器。

目标是：

```text
同一套横屏 HTML
├─ 系统已横屏：直接显示
└─ 系统仍竖屏：交换宽高并旋转 90°
```

参考结构：

```css
html,
body {
  width: 100%;
  height: 100%;
  margin: 0;
  overflow: hidden;
}

.practice-app {
  width: 100vw;
  height: 100dvh;
}

@media (orientation: portrait) and (max-width: 767px) {
  .practice-app {
    position: fixed;
    left: 0;
    top: 100dvh;
    width: 100dvh;
    height: 100vw;
    transform: rotate(-90deg);
    transform-origin: top left;
  }
}
```

说明：

- 该方式用于保持单一横屏布局，不代表学生竖着拿手机阅读。
- 可以在用户点击开始后尝试进入全屏并调用横屏锁定。
- 浏览器横屏锁定只作为增强功能，不能作为页面正常使用的必要条件。
- 锁定失败时应继续使用 CSS 旋转兜底，不显示技术错误。
- 旋转后必须实际检查点击坐标、固定元素、弹层和安全区域。
- 涉及大量文字输入或软键盘的特殊页面，如旋转后明显不稳定，可以在当前任务中单独确认例外方案。

---

### 3.4 主站 iframe 与可视区域

专项练习嵌入主站时，主站和练习页必须共同保证可视区域完整。

原则：

```text
练习 iframe 高度
=
当前可视区域高度
-
主站顶部返回栏或导航栏高度
```

要求：

- 不出现主站与 iframe 的双重纵向滚动。
- iframe 不使用只适合桌面的大型固定高度。
- iframe 外层不能裁切旋转后的手机页面。
- 需要全屏时，应为 iframe 提供相应权限。
- 返回入口应始终清楚，但不能长期占据大量练习空间。
- 若某个浏览器在 iframe 中无法稳定完成全屏、旋转或触控，可在手机端改为顶层页面打开该练习。

普通独立 HTML 不需要自行实现主站导航，但嵌入后必须能够填满主站提供的剩余区域。

---

### 3.5 通用响应式要求

所有 HTML 都必须检查：

- iPad 横屏是否正常。
- 手机真正横屏时是否正常。
- 手机系统仍竖屏时，旋转兜底是否正常。
- 横竖屏切换后布局是否异常。
- 是否出现横向或纵向双重滚动。
- 核心练习区是否完整位于同一屏。
- 文字是否溢出。
- 长单词和长句是否能换行。
- 按钮、词卡和目标区是否容易点击。
- 反馈出现后是否改变练习台位置。
- 页面缩放后是否仍可操作。
- iframe 嵌入后是否被裁切。
- 软键盘弹出后是否影响主要输入区。

建议使用：

```css
* {
  box-sizing: border-box;
}

html,
body {
  max-width: 100%;
  overflow: hidden;
}

.practice-stage {
  min-width: 0;
  min-height: 0;
}

.card,
.option,
.feedback {
  overflow-wrap: anywhere;
}
```

## 4. 视觉风格

### 4.1 默认风格

默认使用：

- 柔和马卡龙配色
- 儿童友好，但不过度幼儿化
- 清楚的文字层级
- 适量圆角
- 轻量阴影
- 紧凑但不拥挤的留白
- 大而清楚的操作按钮
- 柔和但足够清晰的对比度

不默认使用：

- 大面积纯黑
- 大量深灰正文
- 过多高饱和颜色
- 强烈渐变
- 复杂动画
- 商务后台风格
- 每一小块都套多层卡片
- 多个阴影和边框同时争夺注意力

---

### 4.2 专项练习参考配色

专项练习默认延续“特殊疑问词”页面的柔和配色方向。

推荐基础色：

```css
:root {
  --page-bg: #fff4f7;
  --panel-bg: #fffdfc;
  --title: #e88fa3;
  --mint: #a8d8c8;
  --mint-soft: #dff3ea;
  --knowledge: #fff0c9;
  --correct: #d7f3dc;
  --wrong: #ffd9d9;
  --text: #625b63;
  --muted: #8a7f86;
}
```

颜色职责：

- 浅粉：页面背景和整体氛围。
- 暖白：主要练习台。
- 柔和粉：页面标题和少量重点。
- 薄荷绿：主要按钮、栏目标题和完成状态。
- 浅黄：知识点摘要和提示。
- 浅绿：正确。
- 浅红：错误。

规则：

- 不要求所有页面完全使用相同色值，但整体气质和状态含义应保持一致。
- 同一页面只保留一个主要强调色。
- 正确和错误不能只靠颜色表达，还应配合文字、符号或边框。
- 分类区可以使用少量辅助色，但不能让多个颜色同时抢夺注意力。

---

### 4.3 模块化程度

页面可以分区，但不要过度模块化。

允许：

- 顶部信息区
- 核心练习台
- 反馈区
- 必要的题目区和答案区

避免：

- 每句话单独一个大卡片
- 大卡片内再套多层卡片
- 每个知识点一个独立大框
- 过多边框、标签和徽章
- 一屏出现大量互相竞争的颜色
- 用多个纵向面板把页面不断向下撑高

优先通过以下方式分区：

- 对齐
- 留白
- 浅色背景
- 细分隔线
- 标题层级
- 轻微色差

视觉重点应依次放在：

1. 当前可操作内容
2. 操作目标
3. 进度或得分
4. 正误反馈
5. 知识摘要

---

### 4.4 字体和字号

要求：

- 中文和英文均清楚易读。
- 不使用可能导致 IPA 缺字的特殊字体。
- 字号适合 iPad 横屏和手机横屏。
- 题目字号明显大于辅助说明。
- 按钮文字不能太小。
- 英文单词和句子应有足够行高。
- 长文本必须允许换行。
- 手机紧凑模式可以适当缩小，但不能牺牲基本可读性。

建议：

- 页面标题：22–30px
- 题目：18–26px
- 选项：16–21px
- 知识摘要：13–16px
- 辅助说明：13–16px

实际字号应根据内容长度和可用高度调整，不要求机械使用固定数值。

## 5. 交互形式

优先使用：

- 点击选择
- 点击配对
- 点击分类
- 点击排序
- 单选
- 多选
- 选择后提交
- 即时正误反馈
- 逐轮练习
- 全部完成后总结

可以使用拖拽，但不能只依赖拖拽。

### 5.1 点击式替代

任何拖拽练习都应同时提供点击操作。

分类练习：

```text
点击词卡
→ 点击目标分类
→ 完成归类
```

配对练习：

```text
点击左侧项目
→ 点击右侧项目
→ 判断是否配对
```

要求：

- 被选中的项目有明显状态。
- 选择后仍能清楚看到可点击目标。
- 错误后可以继续尝试。
- 不能要求学生在两个相距很远、需要滚动才能到达的区域之间操作。

---

### 5.2 按钮要求

顶部长期保留的按钮应尽量少。

优先保留：

- 检查答案
- 换一组 / 下一轮
- 重新开始

次级功能：

- 显示答案
- 查看详细规则
- 重置当前项目

次级功能可以：

- 在需要时出现
- 放入次级菜单
- 放在完成或检查后显示
- 使用较弱的视觉层级

按钮应：

- 点击区域足够大。
- 文字清楚。
- 不依赖鼠标悬停才能理解。
- 有按下、选中或禁用状态。
- 不挤占核心练习区。
- 在手机横屏紧凑模式中仍然容易点击。

---

### 5.3 正误反馈

反馈应位于固定高度的反馈区，不应因为文字出现而改变练习台的位置。

反馈应：

- 清楚
- 简短
- 及时
- 不羞辱学生
- 不遮挡题目
- 除颜色外同时使用文字或符号

例如：

```text
✓ 全部正确
△ 还差 2 个
× watch 应放到“加 es”
```

错误后可以允许继续尝试，除非当前任务明确要求一次判定。

不要默认立刻显示全部答案。

错误信息较多时：

- 优先显示当前最需要修正的一项。
- 或在反馈区提供简短统计。
- 详细错误可以通过弹层或结果面板查看。
- 不得把多行错误直接向下撑高整个页面。

### 5.4 作答前不得通过视觉设计泄题

练习在学生作答前，不得通过以下视觉因素直接或间接暴露正确答案：

- 特殊颜色；
- 明显不同的大小；
- 固定位置暗示；
- 独有形状或边框；
- 提前出现的动画；
- 只有正确项才具备的装饰或图标。

正常语言线索、题干信息和学生已经学习过的知识可以保留。

学生完成作答后，才可以使用结构颜色、解释文字、动画和结果变化帮助理解答案。

检查时必须确认：学生不依靠视觉规律，也能根据英语知识完成判断。

### 5.5 情境视觉必须参与学习操作

使用火车、小镇、拼图、工厂、侦探等情境时，不能只把普通题型换上装饰背景。

情境中的主要视觉对象应尽量承担真实学习操作，例如：

- 学生移动对象完成排序；
- 点击场景物体完成分类；
- 拼合物体完成配对；
- 使用场景工具定位或修正错误；
- 操作过程直接对应目标语言结构。

情境不是每道练习的强制要求。若使用情境，视觉、操作和知识目标必须相互对应；若无法形成真实对应，应使用更清楚的普通交互，不强行套皮。

## 6. 题型设计原则

### 6.1 分类

适合：

- 词性
- 拼写变化
- 词尾变化
- 音标分类
- 句型分类
- 时间词分类

要求：

- 分类名称清楚。
- 待分类项目与分类目标同时可见。
- iPad 和手机横屏优先使用左右练习台。
- 四个分类目标优先使用 `2 × 2`。
- 支持点击式归类。
- 拖拽只能作为辅助。
- 每轮项目数量应根据可用空间控制。

分类内容较多时，默认拆成多个小回合。一般建议：

- 标准横屏：约 8–10 个待分类项目。
- 手机紧凑横屏：约 6–8 个待分类项目。
- 具体数量以同屏完整、容易点击为准，不机械固定。

不得为了保留全部项目而让词卡区和分类区分开滚动。

---

### 6.2 配对

适合：

- 单词与中文
- 原词与派生词
- 问句与答句
- 单词与图片
- 职业与动作
- 特殊疑问词与用途

要求：

- 左右项目数量一致。
- 每项只存在一个明确答案。
- iPad 和手机横屏均使用左右两列。
- 已配对项目有明显完成状态。
- 左右项目应保持大致对应的行高和节奏。
- 六组左右的短配对可以尽量同屏显示。
- 长句或项目过多时，应减少单轮数量，不通过页面滚动解决。
- 点击左侧后再点击右侧完成配对。

---

### 6.3 选择

适合：

- 单选
- 多选
- 语法判断
- 句子补全
- 音标识别
- 词义判断

要求：

- 题干和全部选项同屏可见。
- 干扰项合理。
- 不使用学生完全没学过的内容作为陷阱。
- 选项长度尽量接近。
- 长选项允许换行。
- 选项数量过多时减少单题选项，而不是增加页面滚动。
- 反馈不遮挡下一次选择。

---

### 6.4 排序

适合：

- 句子排序
- 时间顺序
- 拼写顺序
- 语法结构

要求：

- 除拖拽外提供点击移动。
- 提供当前位置反馈。
- 句子较长时自动换行。
- 全部待排序项目和操作区尽量同屏。
- 项目过多时拆分为多个小题。
- 不使用过小的上移、下移或删除按钮。

---

## 7. 专项练习统一页面骨架

专项练习默认使用“横屏练习台”结构：

```text
顶部紧凑区
标题 + 知识摘要 + 进度/得分 + 主要按钮

核心练习台
占据页面剩余的主要空间

底部反馈条
正确、错误、提示、完成结果
```

推荐 CSS 结构：

```css
.practice-app {
  height: 100dvh;
  display: grid;
  grid-template-rows: auto minmax(0, 1fr) auto;
  overflow: hidden;
}

.practice-stage {
  min-width: 0;
  min-height: 0;
  overflow: hidden;
}
```

### 7.1 顶部紧凑区

顶部可以包含：

- 简短标题
- 一至两行知识摘要
- 进度或得分
- 两到三个主要按钮

要求：

- 不放大段玩法说明。
- 不让每个知识点单独换行。
- 知识点优先使用短行、chip 或紧凑网格。
- 学生进入页面后应立即看到练习内容。
- 顶部高度不能挤压核心练习台。

知识摘要示例：

```text
What 什么｜Who 谁｜Where 哪里｜When 时间
Why 原因｜How 方法｜How old 年龄｜How many 数量
```

```text
直接+s｜大多数动词
加es｜s/x/ch/sh/o结尾
y→ies｜辅音+y
特殊｜have→has，be→is
```

详细规则和大量例词：

- 不默认放在页面底部。
- 可以使用“查看规则”弹层、抽屉或可关闭面板。
- 打开后不能覆盖全部练习内容。
- 关闭后应回到原来的操作位置。

---

### 7.2 核心练习台

核心练习台必须：

- 占据页面剩余的大部分空间。
- 同时显示题目来源和操作目标。
- 不依赖页面纵向滚动完成操作。
- 根据题型使用左右分栏、双列或 `2 × 2` 布局。
- 在内容过多时减少单轮题量。
- 保持点击区域足够大。
- 反馈出现后不改变其尺寸和位置。

允许核心练习台内部使用轻量局部滚动的情况，仅限：

- 非核心补充内容。
- 详细结果清单。
- 当前任务明确允许的特殊长文本。

分类目标、配对双方、题干和答案选项不得因为局部滚动而彼此分离。

---

### 7.3 底部反馈条

底部反馈条用于：

- 当前正误
- 剩余数量
- 简短错误原因
- 完成状态
- 下一步提示

要求：

- 高度稳定。
- 不遮挡练习区。
- 不因文字增加而无限变高。
- 内容过长时优先压缩为摘要，并提供查看详情入口。
- 状态除颜色外同时使用文字或符号。

---

### 7.4 不再默认使用的旧结构

专项练习不再默认采用：

```text
标题
大段说明
多排按钮
独立素材面板
独立互动面板
长错误清单
底部知识总结
```

原因：

- 容易形成单列长页面。
- 核心操作区难以同屏。
- 学生需要反复上下滑动。
- iframe 中更容易出现双重滚动。
- 手机横屏高度不足。

### 7.5 `专项练习通用骨架.html` 的使用边界

生成普通专项练习时，可以优先参考项目文件：

```text
专项练习通用骨架.html
```

可复用的内容包括：

- 横屏页面骨架；
- 顶部紧凑区；
- 核心练习台；
- 固定反馈条；
- 进度显示；
- 横屏与旋转兜底；
- 通用反馈和重置函数。

使用时必须根据具体题型替换核心玩法、题目数据和知识摘要，不能机械复制演示分类题，也不能因为骨架存在而忽略当前任务的明确要求。

---

## 8. 数据和脚本要求

JavaScript 应：

- 结构清楚。
- 不依赖外部库，除非任务明确需要。
- 不产生控制台错误。
- 支持重新开始或换一组。
- 避免重复绑定事件。
- 防止同一道题重复计分。
- 横竖屏切换后仍能正常操作。
- CSS 旋转兜底后点击位置保持正确。
- 反馈不会无限增加页面高度。
- 页面刷新后不要求保留进度，除非任务明确要求。
- 不收集学生信息。
- 不默认连接数据库。

可以在用户点击开始或全屏按钮后尝试：

```js
await document.documentElement.requestFullscreen?.();
await screen.orientation?.lock?.("landscape");
```

但必须：

- 使用错误捕获。
- 失败后继续正常运行。
- 不把方向锁定当作必要条件。
- 不向学生显示浏览器技术错误。
- 保留 CSS 旋转兜底。

生成普通独立练习时，不接入：

- Supabase
- 登录系统
- 学生账号
- 主项目积分
- 外部分析工具

除非当前任务明确要求集成。

# 第二部分：普通练习与 Word 文档

## 10. 普通练习规则

普通练习应根据：

- 用户提供的课堂笔记
- 当前教学进度
- 学生年级
- 学生已学内容
- 用户指定题型

默认要求：

- 不明显低于实际年级。
- 不擅自增加未学内容。
- 每部分题量严格按用户要求。
- 没有指定时，每部分通常控制在 5 题左右。
- 可使用 A / B / C / D / E 分区。
- 题目说明简短清楚。
- 学生版默认不附答案，除非用户要求。

## 10A. 正式测试与周检

正式测试、阶段测验和周检应遵守：

1. 只考已经正式学习并练习过的内容，不为凑题加入未学知识。
2. 以基础巩固为主，但不能简单到只靠猜测即可完成。
3. 英文认中文、基础配对等识别题应清楚直观。
4. 中文写英文、首字母补全和拼写题应保留必要的拼写难度。
5. 题型应有适度变化，减少整页机械重复。
6. 根据学生阶段，可以加入简单句子、短阅读和综合运用，但不得超出已学范围。
7. 较难内容应单独放入“挑战题”或附加题，不影响基础部分得分。
8. 卷面简明、适合打印，并留足书写空间。
9. 图片必须帮助理解题意或完成任务，不能只作装饰，也不能泄露答案。
10. 复习型周检或阶段复习材料可以配一页知识小抄；若当前任务明确为闭卷测验，则不附小抄。
11. 学生版默认不附答案；教师版或答案页按用户要求另行提供。

---

## 11. Word 文档规则

用户要求 Word 时，必须实际生成 `.docx` 文件。

要求：

- 适合打印。
- 标题清楚。
- 题号清楚。
- 答题区足够。
- 页边距合理。
- 不出现内容截断。
- 表格不超出页面。
- 没有要求答案时不附答案。
- 不擅自增加题量。
- 不擅自改变用户保留或删除的题型。

如同一内容同时需要 Word 和 HTML：

- Word 适合打印和书写。
- HTML 应重新设计成交互形式。
- 不直接把纸面排版原样搬进网页。

---

## 12. 普通练习到 HTML 的转换

把普通练习转换为 HTML 时，应重新考虑互动形式。

例如：

- 连线题 → 点击配对
- 分类题 → 点击单词后选择分类
- 选择题 → 大按钮单选
- 排序题 → 点击移动或点击顺序
- 填空题 → 输入框或选词填空
- 圈词题 → 点击高亮目标词

不能只是把 Word 内容截图或静态复制到网页中。

# 第三部分：整理英语校内笔记

## 13. 默认触发

当用户说：

> 整理英语校内笔记

默认按照本部分规则处理。

---

## 14. Markdown 主文档

每个单元的主文档可以包括：

1. 单元或主题概览
2. 来源信息
3. 学习目标
4. 课时或模块拆分
5. 分类词汇
6. 固定搭配
7. 重点句子
8. 语法与关键句型
9. 易错点
10. 复习建议
11. 可转换成练习的内容

不要求每个单元机械拥有完全相同的章节，但整体结构应统一。

## 15. 机器可读文件

根据材料情况，可以生成：

- `*.wordbank.csv`
- `*.summary.json`

### wordbank.csv

可包含：

- 单词
- 中文义
- 词性
- 所属单元
- 所属课时
- 分类
- 是否核心词
- 备注

具体字段应根据当前材料需要确定，不随意增加无用字段。

### summary.json

可记录：

- 单元主题
- 学习目标
- 重点语法
- 重点句型
- 词汇分类
- 练习建议

## 16. 多单元处理

处理多个单元时：

1. 每个单元单独整理。
2. 检查每个单元是否完整。
3. 生成单元独立文件。
4. 根据用户要求生成合并版。
5. 最后检查是否漏单元。
6. 需要时生成 ZIP 压缩包。

任务过大时可以拆批，但最终必须完成合并和检查。

---

## 17. 来源文件处理

### DOCX

- 纯文字和表格可以直接整理。
- 保留标题层级和表格中的教学信息。
- 不把页眉页脚等无关内容混入知识点。

### PDF

- 需要检查文本和页面结构。
- 有图表、截图或复杂排版时，应结合页面视觉内容分析。
- 不只依赖文本提取结果判断版面信息。

### 图片和扫描件

- 需要进行视觉识别。
- 手写或低清内容存在识别风险。
- 无法确认的内容应明确标记，不能猜测。
